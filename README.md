# aqnas-studio

Command center for the AQNAS software studio — a one-person, Claude Code powered development environment.

Designed to run on **macOS** or **Linux (Ubuntu 24.04 or similar)**. The production server is assumed to be Ubuntu 24.04 but the studio itself is environment-agnostic.

## What's here

- `claude-config/` — C-level agents, commands, and skills for Claude Code. Gets symlinked to `~/.claude/` during setup.
- `infrastructure/server/` — Caddy, systemd, and operational scripts for the production server. Works on any Ubuntu 24.04 host (cloud VM, bare metal, even a Raspberry Pi).
- `meetings/` — Studio-wide meeting outputs. Gitignored — cross-project strategy stays private.
- `docs/` — Studio plan and documentation.
- `setup.sh` — one-shot installer for symlinks, env var, and pre-commit hook.

## Setup

### 1. Install the tools you'll need

| Tool | Install command |
|---|---|
| **Claude Code** | Follow https://docs.claude.com/en/docs/claude-code |
| **git** | macOS: `brew install git` · Ubuntu: `sudo apt install git` |
| **uv** | `curl -LsSf https://astral.sh/uv/install.sh \| sh` |
| **gitleaks** | macOS: `brew install gitleaks` · Ubuntu: `sudo apt install gitleaks` |

### 2. Clone the repo

```sh
mkdir -p ~/dev
git clone git@github.com:yjaquinas/aqnas-studio.git ~/dev/aqnas-studio
cd ~/dev/aqnas-studio
```

`~/dev/aqnas-studio` is the recommended path — keeps code separate from your personal files. If you prefer a different location (and know your way around shell env vars), clone wherever you want; `setup.sh` handles the path automatically.

### 3. Run the setup script

```sh
./setup.sh
```

This one script handles:
- Backing up your existing `~/.claude/` (if any) and creating the symlink
- Setting `AQNAS_STUDIO_ROOT` in your shell rc file
- Installing the gitleaks pre-commit hook in this repo
- Checking that the required tools are installed

The script is idempotent — safe to re-run if you move the repo or want to refresh the setup.

### 4. Activate the environment variable in your current shell

The script adds `AQNAS_STUDIO_ROOT` to your rc file, but the current shell session doesn't know about it yet. Either:

```sh
# Option A — reload your shell config
source ~/.zshrc          # or ~/.bashrc if you use bash

# Option B — open a fresh terminal (easiest)
```

The script tells you the exact `export` line to copy-paste if you'd rather do it manually.

### 5. Verify it's working

Open Claude Code:

```sh
claude
```

Type `/` and confirm you see: `/run-meeting`, `/start-new-app`, `/commit-git`, plus a few others. If you do, setup worked.

### 6. Configure SSH access to the production server (one-time, per dev machine)

If you don't yet have a production server, skip this step — come back when you do.

The studio uses an SSH alias so the production server's IP never appears in commands, scripts, or docs. The alias lives in your `~/.ssh/config`, never in the repo.

Add this block (substituting your server's IP or DNS name):

```sh
cat >> ~/.ssh/config <<'EOF'

Host aqnas-prod
    HostName <YOUR_SERVER_IP_OR_DNS>
    User ubuntu
    IdentityFile ~/.ssh/id_ed25519
    StrictHostKeyChecking accept-new
EOF

chmod 600 ~/.ssh/config
```

Verify:

```sh
ssh aqnas-prod whoami
# Expected: ubuntu
```

Studio docs, the `MANUAL-TASKS.md` checklists generated by `/start-new-app`, and the server-side scripts in `infrastructure/server/scripts/` all reference `aqnas-prod`. Once configured, you never need to remember (or type) the IP again.

> **If you already have an alias for this server** (e.g. from prior setup), add `aqnas-prod` as a second host name on the same `Host` line instead of duplicating the block:
> ```
> Host existing-alias aqnas-prod
>     HostName ...
>     ...
> ```
> SSH treats them as the same host under two names. Both keep working. Your muscle memory stays intact.

**Important:** the alias only works on this dev machine. GitHub Actions runners don't read your local SSH config, so the `SSH_HOST` GitHub secret still uses the raw IP or DNS name. The split is intentional — humans use the alias, CI/CD uses the configured secret.

### Troubleshooting

- **No studio commands in the `/` picker.** The symlink isn't working. Run `readlink ~/.claude` — it should print a path ending in `claude-config`. If it prints nothing, re-run `./setup.sh`.
- **Lost my MCP settings.** Setup moved your old `~/.claude/` to `~/.claude.backup-*`. Copy any settings you need from there to the new location.
- **Commands show up but error when run.** Your Claude Code might be too old. Update it from https://docs.claude.com/en/docs/claude-code.
- **`AQNAS_STUDIO_ROOT` isn't set.** Open a new terminal, or run `source ~/.zshrc` (or `~/.bashrc`). If still unset, check that `./setup.sh` finished without errors.
- **`ssh aqnas-prod` says "Could not resolve hostname" or hangs.** Either the alias isn't in your `~/.ssh/config`, or the `HostName` value is wrong. Verify with `ssh -G aqnas-prod | grep hostname`. The output should show the resolved IP or DNS name.

## How to use

The studio runs on a simple 4-step loop: **meeting → update config → build → commit.** Here's how each one looks.

> Looking for deploy, server, status, or project-conversion workflows? See [`docs/CHEATSHEET.md`](docs/CHEATSHEET.md) for the project lifecycle cheat sheet covering new projects, ongoing development, and converting existing projects to studio convention.

### Start a brand-new project

```sh
mkdir ~/dev/my-project
cd ~/dev/my-project
claude
```

Then inside Claude Code:

```
/start-new-app
```

That's it. The skill scaffolds the current directory in place — populates the canonical project layout, reserves a port from the studio registry, runs `uv sync`, initializes git with a gitleaks pre-commit hook, and makes the initial commit.

Optional flags:
- `--no-mobile` — skip the Hyperview mobile templates and `mobile-client/` directory
- `--no-web` — skip the HTMX web templates (rare; mobile-only setups)

Default scaffolds both web and mobile layers.

After scaffolding, the skill prints next-step instructions: how to run the project locally, how to push to GitHub, and what's needed on the production server.

### Add a feature to an existing project

```sh
cd ~/{project-name}
claude
```

Then inside Claude Code:

```
/run-meeting <what you want to build>
/update-project-claude meetings/MEETING-*/2-CLAUDE-UPDATES.md
/exit
claude --continue
/execute-plan meetings/MEETING-*/3-BUILD-PLAN.md
/commit-git
```

The `/exit` and `claude --continue` in the middle is important — the session has to restart so new agents and skills get loaded before the build runs.

### Just code along without a meeting

For small changes you don't need a full meeting for, just talk to Claude like normal and commit when you're done:

```
/commit-git
```

`/commit-git` scans for secrets and writes a conventional commit message — use it instead of `git commit` directly.

### The slash commands cheatsheet

| Command | What it does |
|---|---|
| `/run-meeting <topic>` | Agents deliberate, produce meeting outputs |
| `/run-meeting mode-auto-on <topic>` | Same, but agents iterate without asking you much |
| `/start-new-app [--no-web] [--no-mobile]` | Scaffold the current directory into a new AQNAS project |
| `/update-project-claude <updates-file>` | Apply `.claude/` changes from a meeting |
| `/execute-plan <build-plan-file>` | Run the build-plan tasks |
| `/commit-git` | Safe, secret-scanned git commit |
| `/analyze-webpage <url>` | Study a reference site (design, copy, stack) |

## Working across multiple machines

The repo is the source of truth. On machine A:

```sh
cd $AQNAS_STUDIO_ROOT
git add claude-config/
git commit -m "..."
git push
```

On machine B (after running setup.sh there too):

```sh
cd $AQNAS_STUDIO_ROOT
git pull
```

Changes take effect next time you start Claude Code.
